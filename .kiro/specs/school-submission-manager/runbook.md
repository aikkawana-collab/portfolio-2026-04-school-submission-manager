## 1. Severity定義

| Severity | 定義 | 応答目標 | 例 |
|----------|------|---------|-----|
| Sev1 (緊急) | 全ユーザーがシステムを利用不可 | 15分以内に対応開始 | Webアプリが完全にエラー |
| Sev2 (高) | 主要機能が利用不可 | 1時間以内に対応開始 | フォーム→SS連携の停止 |
| Sev3 (中) | 一部機能に問題あり | 当日中に対応 | 表示が崩れる、遅延が大きい |
| Sev4 (低) | 軽微な不具合 | 次回メンテナンス時に対応 | UIの微細な表示不良 |

## 2. シナリオ別対応手順

### Scenario 1: Webアプリがエラーで表示されない (Sev1)

**症状**: ブラウザでWebアプリURLにアクセスすると「スクリプトでエラーが発生しました」と表示される

**対応手順**:
1. Google Apps Scriptのダッシュボード (script.google.com) にアクセス
2. 「実行数」からエラーログを確認
3. エラー内容を特定:
   - 権限エラー → 手順3aへ
   - スクリプトエラー → 手順3bへ
   - クォータ超過 → 手順3cへ

**3a. 権限エラーの場合**:
- デプロイ設定を確認（「自分として実行」が設定されているか）
- 再デプロイを実行

**3b. スクリプトエラーの場合**:
- エラーのスタックトレースを確認
- 直前のデプロイに問題がある場合、前バージョンにロールバック:
  - GASダッシュボード → 「デプロイ」→「デプロイを管理」→ 前バージョンを選択

**3c. クォータ超過の場合**:
- 24時間後にクォータがリセットされるのを待つ
- 緊急時: スプレッドシートを直接閲覧して代替運用

### Scenario 2: フォーム送信がスプレッドシートに反映されない (Sev2)

**症状**: フォーム送信後、スプレッドシートのステータスが更新されない

**対応手順**:
1. フォームの回答スプレッドシートで回答が記録されているか確認
   - 記録されていない → Googleフォームの連携設定を確認
   - 記録されている → トリガーの問題。手順2へ
2. GASダッシュボード → 「トリガー」を開く
3. onFormSubmitトリガーが存在するか確認
   - 存在しない → トリガーを再作成
   - 存在する → 「実行数」でエラーを確認
4. トリガーの再作成手順:
   - GASエディタ → 「トリガー」→「+ トリガーを追加」
   - 関数: onFormSubmitHandler
   - イベントソース: スプレッドシートから
   - イベントタイプ: フォーム送信時

### Scenario 3: 児童用画面で他の児童のデータが表示される (Sev1)

**症状**: 児童Aがアクセスした際に、児童Bのデータが表示される

**対応手順**:
1. **即座にWebアプリを無効化**: デプロイ設定で一時的にアクセスを制限
2. UUID識別子のマッピングを「名簿」シートで確認
3. AuthServiceのロール判定ロジックを確認
4. 修正後、テスト環境 (/dev) で検証してから再デプロイ

### Scenario 4: 応答が極端に遅い (Sev3)

**症状**: Webアプリの表示に10秒以上かかる

**対応手順**:
1. GASダッシュボード → 「実行数」で実行時間を確認
2. キャッシュの状態を確認:
   - GASエディタで `CacheService.getScriptCache().get('allStudentData')` を実行
   - null の場合、キャッシュが無効化されている
3. 手動でキャッシュを再構築: invalidateCache() 関数を手動実行
4. 改善しない場合: スプレッドシートのデータ量を確認（不要なシートや古いデータの整理）

## 3. ロールバック手順

### GASデプロイのロールバック

1. script.google.com でプロジェクトを開く
2. 「デプロイ」→「デプロイを管理」
3. デプロイ一覧から、正常動作していたバージョンを特定
4. 該当バージョンの「アクティブ」をオンにする
5. ブラウザで動作確認

### claspによるロールバック（開発者向け）

```bash
# Gitで前バージョンに戻す
git log --oneline
git checkout <commit-hash> -- src/

# GASにプッシュ
clasp push

# 新しいデプロイを作成
clasp deploy --description "Rollback to v<version>"
```

## 4. 定期メンテナンス

| 頻度 | 作業 | 手順 |
|------|------|------|
| 毎週 | 実行ログの確認 | GASダッシュボード → 「実行数」でエラーがないか確認 |
| 毎月 | キャッシュ動作の確認 | SLI-004 キャッシュヒット率が80%以上か確認 |
| 年1回 | 年度更新 | 「名簿」シートの児童データを新年度に差替え、UUID再生成 |
| 年1回 | GASバージョン確認 | GASランタイムの更新がないか確認 |

## 5. 緊急時の代替運用

システムが完全に利用不可の場合の代替手段:

| 機能 | 代替手段 |
|------|---------|
| ステータス入力 | スプレッドシートに直接入力 |
| 全体名簿閲覧 | スプレッドシートを直接閲覧（条件付き書式は維持） |
| 教師ダッシュボード | スプレッドシートのフィルタ機能で代替 |
| 児童確認画面 | 教師が口頭で未提出課題を伝達 |
