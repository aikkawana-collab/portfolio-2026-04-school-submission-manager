## 1. プロジェクト概要

### 1.1 背景と課題

小学校において、提出物の管理は紙ベースの名簿で行われている。この運用には以下の構造的課題がある。

| # | 課題 | 影響 | 影響度 |
|---|------|------|--------|
| P1 | 転記ミス | 提出済の課題が未提出と記録され、児童・保護者への誤った指導が発生する | 高 |
| P2 | リアルタイム性の欠如 | 提出状況の把握に時間差が生じ、即時のフォローアップができない | 高 |
| P3 | 児童自身による進捗確認の困難さ | 自律的な学習管理が育まれず、教師への確認依頼が頻発する | 中 |
| P4 | 複数クラス・課題の横断管理の煩雑さ | 教師の事務作業時間が増大し、本来の教育業務を圧迫する | 高 |

### 1.2 プロジェクトの目的

Google Workspace（Googleフォーム・Googleスプレッドシート・Google Apps Script）を基盤とした統合的な提出物管理ツールを構築し、上記課題を解決する。

### 1.3 スコープ

#### In Scope（対象範囲）

- Googleフォームによる提出物ステータスの入力
- スプレッドシートへの自動反映と全体名簿の管理
- 教師向けダッシュボード（GAS Webアプリ）
- 児童向け未提出確認画面（GAS Webアプリ）
- 条件付き書式によるスプレッドシートの視認性向上

#### Out of Scope（対象外）

- 成績管理・通知表との連携
- 保護者向けポータル画面（将来拡張として定義）
- 他校・自治体レベルでの統合管理
- 外部学習教材プラットフォームとの自動連携

### 1.4 ステークホルダー

| ステークホルダー | 役割 | 主な関心事 |
|------------------|------|-----------|
| 担任教師 | 主要ユーザー・管理者 | 提出状況の即時把握、事務作業の削減 |
| 児童（小学1〜6年生） | エンドユーザー | 自分の未提出課題の簡単な確認 |
| 学校管理者（教頭・校長） | 承認者 | セキュリティ、コスト、運用負荷 |
| Google Workspace管理者 | デプロイ担当 | GASデプロイ権限、アカウント管理 |
| 保護者 | 間接的ステークホルダー | 子供の学習状況の透明性（将来拡張） |

### 1.5 用語定義

| 用語 | 定義 |
|------|------|
| 提出物 | 教師が児童に対して提出を求める課題・プリント・宿題の総称 |
| ステータス | 提出物の状態。「未提出」「提出済」「再提出」「確認済」のいずれか |
| 全体名簿 | 全児童×全提出物のマトリクス形式のスプレッドシート |
| ダッシュボード | 教師がブラウザで閲覧する個人別の提出状況画面 |
| 確認画面 | 児童がタブレットで閲覧する自分の未提出課題一覧画面 |

## 2. 機能要件

### 2.1 データ入力・自動反映

#### REQ-FUNC-001: フォーム送信による提出物ステータスの記録

**EARS形式**: When 教師がGoogleフォームで提出物ステータスを送信する, the システムは送信データをスプレッドシートの該当セルに自動反映する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must (MoSCoW) |
| トリガー | Googleフォーム送信イベント |
| 入力 | 児童識別子（出席番号）、提出物名、ステータス（提出済/再提出/未提出）、教師コメント（任意）、タイムスタンプ |
| 出力 | スプレッドシートの該当行・列にステータスとタイムスタンプが記録される |
| 受入基準 | フォーム送信後30秒以内にスプレッドシートに反映されること |

#### REQ-FUNC-002: インストーラブルトリガーによるイベント駆動処理

**EARS形式**: When フォーム送信イベント（onFormSubmit）が発火する, the システムはインストーラブルトリガー経由でGAS関数を実行し、データの検証・変換・書き込みを行う.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 技術制約 | Simple Triggersは権限制限があるため使用しない。インストーラブルトリガーを使用する |
| 処理内容 | e.namedValues からデータ取得 → 児童識別子で対象行を検索 → ステータス列を更新 → タイムスタンプ記録 |

#### REQ-FUNC-003: 複数提出物の一括入力

**EARS形式**: When 教師が1回のフォーム送信で複数の提出物ステータスを入力する, the システムは各提出物ごとにスプレッドシートの対応列を個別に更新する.

| 属性 | 値 |
|------|-----|
| 優先度 | Should |
| 受入基準 | 1回のフォーム送信で最大10件の提出物ステータスを同時に記録できること |

### 2.2 全体名簿（スプレッドシート）

#### REQ-FUNC-010: 全体名簿の自動構成

**EARS形式**: While システムが稼働している間, the 全体名簿シートは行に児童（出席番号順）、列に提出物名を配置したマトリクス構造を維持する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| データ構造 | 行: 児童（A列: 出席番号, B列: 氏名, C列以降: 各提出物のステータス） |
| 受入基準 | 新しい提出物が追加された場合、列が自動的に追加されること |

#### REQ-FUNC-011: 条件付き書式による視認性の確保

**EARS形式**: While 全体名簿が表示されている間, the システムは提出物ステータスに応じた条件付き書式を自動適用し、未提出・再提出・提出済を色分け表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 実装方式 | ConditionalFormatRuleBuilder によるプログラム的制御 |
| 書式ルール | 未提出（空セル）→ 赤背景(#FF0000), 再提出 → 橙背景(#FFA500), 提出済 → 緑背景(#4CAF50), 確認済 → 青背景(#2196F3) |
| 受入基準 | 教師がシートを開いた瞬間に、未提出箇所が直感的に識別できること |

#### REQ-FUNC-012: 提出率の自動計算

**EARS形式**: While 全体名簿が表示されている間, the システムは各児童の提出率（提出済件数÷全課題数×100）を自動計算し、最終列に表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 表示形式 | パーセンテージ表示 + グラデーション条件付き書式（赤〜黄〜緑） |
| 受入基準 | 提出率が正確に計算され、リアルタイムで更新されること |

#### REQ-FUNC-013: 期限切れ提出物のハイライト

**EARS形式**: When 提出物の締切日を過ぎてもステータスが「未提出」のままである場合, the システムは該当セルを強調表示（太字+赤枠線）する.

| 属性 | 値 |
|------|-----|
| 優先度 | Should |
| 実装方式 | whenFormulaSatisfied() によるカスタム数式評価 |

### 2.3 教師向けダッシュボード（Webアプリ）

#### REQ-FUNC-020: 教師用ダッシュボードの提供

**EARS形式**: When 教師がWebアプリにアクセスし ?view=teacher パラメータが指定されている場合, the システムは教師用ダッシュボード画面を表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 認証 | Googleアカウント認証（学校ドメイン） |
| アクセス方法 | GAS WebアプリURL + ?view=teacher |

#### REQ-FUNC-021: 児童別の提出状況詳細表示

**EARS形式**: When 教師がダッシュボード上で児童を選択する, the システムは当該児童の全提出物ステータス、提出率、未提出一覧、再提出一覧を表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 表示項目 | 児童氏名、出席番号、提出率（円グラフ）、未提出課題一覧、再提出課題一覧、提出履歴タイムライン |
| 受入基準 | 3クリック以内で任意の児童の詳細を確認できること |

#### REQ-FUNC-022: クラス全体の統計表示

**EARS形式**: While 教師用ダッシュボードが表示されている間, the システムはクラス全体の提出率平均、未提出者数、再提出対象者数をサマリー表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 表示項目 | クラス平均提出率、未提出者一覧（人数と氏名）、再提出対象者一覧、課題別提出率 |

#### REQ-FUNC-023: フィルタリング・検索機能

**EARS形式**: When 教師がダッシュボード上でフィルタ条件（未提出のみ/再提出のみ/特定課題）を指定する, the システムは条件に合致する児童の一覧のみを表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Should |
| フィルタ項目 | ステータス別、課題別、提出率範囲、氏名検索 |

#### REQ-FUNC-024: CSVエクスポート

**EARS形式**: When 教師がダッシュボード上で「エクスポート」ボタンを押す, the システムは現在表示中のデータをCSV形式でダウンロードさせる.

| 属性 | 値 |
|------|-----|
| 優先度 | Could |

### 2.4 児童向け未提出確認画面（Webアプリ）

#### REQ-FUNC-030: 児童用確認画面の提供

**EARS形式**: When 児童がWebアプリにアクセスし ?view=student&id={出席番号} パラメータが指定されている場合, the システムは当該児童の未提出確認画面を表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| アクセス方法 | GAS WebアプリURL + ?view=student&id={出席番号} |
| デバイス | タブレット端末（iPadまたはChromebook） |

#### REQ-FUNC-031: 未提出課題の一覧表示

**EARS形式**: While 児童用確認画面が表示されている間, the システムは当該児童の未提出課題と再提出課題の一覧を、大きな文字とアイコンで表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 表示項目 | 課題名、ステータス（アイコン付き）、締切日（該当する場合） |
| UI要件 | UDフォント、文字サイズ16px以上、タップ領域44×44px以上 |

#### REQ-FUNC-032: 提出率の視覚的表示

**EARS形式**: While 児童用確認画面が表示されている間, the システムは当該児童の提出率をプログレスバーと数値で表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 表示形式 | プログレスバー（達成率を色で段階表示）+ 「全20回中16回提出済（80%）」のような具体的テキスト |

#### REQ-FUNC-033: 全課題提出完了時の達成フィードバック

**EARS形式**: When 児童の全課題が「提出済」または「確認済」になった場合, the システムは達成アニメーションとバッジを表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Should |
| 表示内容 | 完了アニメーション（紙吹雪エフェクト等）、達成バッジアイコン、「よくがんばりました！」メッセージ |

#### REQ-FUNC-034: 教師コメントの表示

**EARS形式**: When 教師が特定の提出物に対してコメントを記録している場合, the システムは児童用確認画面にコメントを表示する.

| 属性 | 値 |
|------|-----|
| 優先度 | Should |
| 制約 | 心理的安全性に配慮し、否定的表現を含まない形式でコメントを表示する。教師入力時にガイドラインをUIに表示する |

## 3. 非機能要件

### 3.1 性能要件

#### REQ-NFR-001: フォーム送信からの反映時間

**EARS形式**: When 教師がGoogleフォームを送信する, the システムはスプレッドシートへの反映を30秒以内に完了する.

| 属性 | 値 |
|------|-----|
| 指標 | フォーム送信〜スプレッドシート反映の遅延時間 |
| 目標値 | 30秒以内（p95） |
| 計測方法 | トリガー実行ログのタイムスタンプ差分 |

#### REQ-NFR-002: Webアプリの初期表示時間

**EARS形式**: When ユーザーがWebアプリにアクセスする, the システムは画面の初期表示を5秒以内に完了する.

| 属性 | 値 |
|------|-----|
| 指標 | First Contentful Paint (FCP) |
| 目標値 | 5秒以内（p95） |
| 前提条件 | キャッシュヒット時。キャッシュミス時は8秒以内 |

#### REQ-NFR-003: 同時アクセス耐性

**EARS形式**: While 朝のホームルーム時に全児童（最大40名）が同時にアクセスする場合, the システムはCacheServiceを利用してスプレッドシートAPIコールを最小化し、全ユーザーに対して応答する.

| 属性 | 値 |
|------|-----|
| 同時アクセス数 | 最大40名/クラス |
| キャッシュ戦略 | CacheService.getScriptCache()、TTL: 600秒 |
| 受入基準 | 40名同時アクセス時にエラーが発生しないこと |

#### REQ-NFR-004: GAS実行時間制限の遵守

**EARS形式**: While GAS関数が実行されている間, the システムは単一の関数実行を6分以内に完了する.

| 属性 | 値 |
|------|-----|
| 制約 | Google Apps Scriptの実行時間制限: 6分/実行 |
| 対策 | バッチ処理の分割、不要なAPI呼び出しの排除、キャッシュの活用 |

### 3.2 セキュリティ要件

#### REQ-SEC-001: Googleアカウントによる認証

**EARS形式**: When ユーザーがWebアプリにアクセスする, the システムはGoogleアカウントのOAuth認証を要求し、認証済みユーザーのみにアクセスを許可する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 認証方式 | Google OAuth 2.0（学校ドメイン制限） |
| デプロイ設定 | 「次のユーザーとして実行: 自分」「アクセスできるユーザー: ドメイン内の全ユーザー」 |

#### REQ-SEC-002: ロールベースのアクセス制御

**EARS形式**: When ユーザーがWebアプリにアクセスする, the システムはユーザーのロール（教師/児童）に基づいて表示可能な情報を制限する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 教師ロール | 全児童のデータ閲覧・編集が可能 |
| 児童ロール | 自分のデータのみ閲覧可能。他の児童のデータにはアクセスできない |
| 実装方式 | サーバーサイド（GAS）でのロール判定。クライアントサイドのみでの制御は不可 |

#### REQ-SEC-003: スクリプトインジェクション防止

**EARS形式**: When ユーザー入力をHTML画面に表示する, the システムは全ての入力値をサニタイズし、XSS（クロスサイトスクリプティング）攻撃を防止する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 対策 | HTMLエスケープ処理、Content Security Policy (CSP) ヘッダーの適用、テンプレートリテラルでのユーザー入力の直接埋め込み禁止 |

#### REQ-SEC-004: データのドメイン内完結

**EARS形式**: While システムが稼働している間, the 全てのデータはGoogleのインフラストラクチャ内に保持され、外部サービスへのデータ送信は行わない.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 対象 | 児童の個人情報（氏名、出席番号）、提出状況データ |

#### REQ-SEC-005: 児童IDの推測困難性

**EARS形式**: When 児童用確認画面のURLにidパラメータが使用される, the システムは連番の出席番号ではなく、推測困難な識別子を使用する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 実装方式 | UUID v4 または ハッシュ化された識別子をスプレッドシートの隠し列に保持 |
| 理由 | 連番IDでは ?id=1, ?id=2 と試行することで他児童のデータにアクセスされるリスクがある |

### 3.3 可用性要件

#### REQ-NFR-010: システム稼働時間

**EARS形式**: While 学校の授業日（平日8:00-16:00）の間, the システムは99.5%以上の可用性を維持する.

| 属性 | 値 |
|------|-----|
| 目標 | 99.5%（月間最大ダウンタイム: 約22分） |
| 依存 | Google Workspaceの稼働率に依存 |

#### REQ-NFR-011: エラー時のフォールバック

**EARS形式**: If Webアプリの読み込みに失敗した場合, the システムは「読み込み中です。しばらくお待ちください」メッセージを表示し、自動リトライを行う.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| リトライ | 最大3回、5秒間隔 |

### 3.4 ユーザビリティ要件

#### REQ-NFR-020: 児童向けUI — 文科省ガイドライン準拠

**EARS形式**: While 児童用確認画面が表示されている間, the システムは文部科学省「ICT活用による児童生徒の健康面への影響等に関する配慮」ガイドラインに準拠したUI表示を行う.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 背景色 | オフホワイト (#F5F5F0) を基本とし、真っ白(#FFFFFF)は使用しない |
| コントラスト比 | WCAG 2.1 AA基準（4.5:1以上）を満たすこと |
| フォント | UDデジタル教科書体またはBIZ UDゴシック |
| 最小文字サイズ | 16px |
| タップ領域 | 44×44px以上 |

#### REQ-NFR-021: ダークモード対応

**EARS形式**: When 児童の端末がダークモード設定になっている場合, the システムは自動的にダークモードの配色に切り替える.

| 属性 | 値 |
|------|-----|
| 優先度 | Could |
| 実装方式 | prefers-color-scheme メディアクエリ |

#### REQ-NFR-022: タブレット端末への最適化

**EARS形式**: While 児童がタブレット端末（iPad / Chromebook）でアクセスする場合, the システムはレスポンシブデザインにより画面サイズに応じた最適なレイアウトを提供する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 対応画面幅 | 768px〜1366px（タブレット横持ち・縦持ち両対応） |

#### REQ-NFR-023: 教師向けUI — 操作効率

**EARS形式**: When 教師がダッシュボードを使用する場合, the システムは主要操作を3クリック以内で完了できるUI構造を提供する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 対象操作 | 児童選択、フィルタリング、詳細表示 |

### 3.5 保守性要件

#### REQ-NFR-030: 年度更新への対応

**EARS形式**: When 年度が変わり新しいクラス名簿が作成される場合, the システムはスプレッドシートの名簿データを差し替えるだけで新年度の運用を開始できる.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 受入基準 | 年度更新作業が30分以内に完了すること。コード変更を必要としないこと |

#### REQ-NFR-031: 新規提出物の追加容易性

**EARS形式**: When 教師が新しい提出物をGoogleフォームに追加する場合, the システムはスプレッドシートとWebアプリに自動的に反映する.

| 属性 | 値 |
|------|-----|
| 優先度 | Must |
| 受入基準 | フォームの選択肢追加のみで新規提出物が運用に組み込まれること |
