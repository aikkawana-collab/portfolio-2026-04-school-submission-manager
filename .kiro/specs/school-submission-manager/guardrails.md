## 1. 権限境界定義

### 1.1 Claude Code の操作権限

| 操作 | 許可 | 条件 |
|------|------|------|
| ローカルファイルの読み書き | 許可 | src/ 配下のみ |
| clasp push（GASへのアップロード） | 許可 | テスト環境 (/dev) のみ自動実行 |
| clasp deploy（本番デプロイ） | 要承認 | 人間の確認後のみ実行 |
| スプレッドシートの構造変更 | 要承認 | 既存データに影響する変更は確認必須 |
| トリガーの作成・削除 | 要承認 | 権限変更を伴うため |
| 児童の個人情報へのアクセス | 禁止 | テスト用のダミーデータのみ使用 |

### 1.2 GASスクリプトの実行権限

| 操作 | 教師ロール | 児童ロール |
|------|----------|----------|
| 全児童データの読み取り | 許可 | 禁止 |
| 自分のデータの読み取り | 許可 | 許可 |
| ステータスの書き込み | 許可（フォーム経由） | 禁止 |
| 設定の変更 | 許可 | 禁止 |
| キャッシュの無効化 | 許可 | 禁止 |

## 2. Human-in-the-Loop ゲート

### 2.1 開発フェーズのゲート

| ゲート | タイミング | 承認者 | チェック項目 |
|--------|----------|--------|------------|
| G1: 設計レビュー | Phase 1 開始前 | 開発者 | requirements.md との整合性 |
| G2: セキュリティレビュー | Phase 1 完了後 | 開発者 | STRIDE脅威モデルの緩和策が全て実装されているか |
| G3: UI/UXレビュー | Phase 2 完了後 | 教師（実際のユーザー） | 文科省ガイドライン準拠、操作性 |
| G4: 本番デプロイ承認 | Phase 3 完了後 | システム管理教師 | テスト結果の確認 |
| G5: 年度更新確認 | 年次 | システム管理教師 | 新名簿データの正確性 |

### 2.2 運用フェーズのゲート

| ゲート | タイミング | 承認者 | チェック項目 |
|--------|----------|--------|------------|
| G6: 新規提出物の追加 | 随時 | 教師 | フォーム・SS連携の動作確認 |
| G7: バージョンアップ | 随時 | システム管理教師 + 開発者 | テスト環境での検証完了 |
| G8: 障害時のロールバック | 緊急時 | システム管理教師 | 前バージョンの動作確認 |

## 3. 監査証跡要件

### 3.1 記録対象イベント

| イベント | 記録先 | 記録項目 |
|---------|--------|---------|
| フォーム送信 | 「提出履歴」シート | タイムスタンプ、教師メール、児童番号、変更内容 |
| Webアプリアクセス | GAS実行ログ | タイムスタンプ、ユーザーメール、アクセスパラメータ |
| エラー発生 | GAS実行ログ | タイムスタンプ、エラー内容、スタックトレース |
| デプロイ | GASバージョン履歴 | バージョン番号、デプロイ日時、説明 |

### 3.2 ログ保持期間

| ログ種別 | 保持期間 | 備考 |
|---------|---------|------|
| 提出履歴 | 1年度分（年度末にアーカイブ） | スプレッドシート容量を考慮 |
| GAS実行ログ | Google側の保持期間に依存（約7日） | 重要なエラーは別途記録 |
| デプロイ履歴 | 永続 | GASプラットフォームが管理 |

## 4. データ保護ガードレール

### 4.1 個人情報の取り扱い

| ルール | 詳細 |
|--------|------|
| 最小権限の原則 | 児童用画面には自分のデータのみ返却。全体データをクライアントに送信しない |
| データの局所性 | 全データはGoogleインフラ内に保持。外部サービスへの送信禁止 |
| テストデータ | 開発時は実際の児童名ではなくダミーデータを使用 |
| アクセスログ | 誰がいつどのデータにアクセスしたかを記録 |

### 4.2 教師コメントの安全性

| ルール | 詳細 |
|--------|------|
| UI上のガイドライン表示 | コメント入力欄に「前向きな表現を心がけてください」等のガイドを表示 |
| 文字数制限 | コメントは100文字以内に制限 |
| 禁止ワードチェック | 将来拡張として、否定的な表現のチェック機能を検討 |

## 5. AI開発時のガードレール（Claude Code使用時）

### 5.1 コード生成のルール

| ルール | 詳細 |
|--------|------|
| Read before Write | 既存ファイルを変更する前に必ず読み取る |
| 1変更1目的 | 1回のコード変更は1つの目的に限定する |
| セキュリティ最優先 | ロール検証のない関数を公開しない |
| テスト必須 | 新規関数には必ずテストコードを併記 |

### 5.2 禁止事項

| 禁止事項 | 理由 |
|---------|------|
| 児童の実データを使ったテスト | 個人情報保護 |
| eval() の使用 | スクリプトインジェクションのリスク |
| 外部URLへのfetch | データの外部流出防止 |
| UrlFetchApp.fetch() の無制限使用 | APIクォータの保護 |
| clasp deploy の自動実行 | 本番環境の保護 |
